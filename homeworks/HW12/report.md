# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv` с часовыми временными рядами, прогнозировалось значение `target`
- Организован корректный `train / validation / test` split по времени в пропорции 70%/15%/15%
- Сравнивались 4 подхода: B1 (naive-last), B2 (moving-average), B3 (ridge-lag-features), R1 (gru-forecast)
- Лучшим оказался GRU подход благодаря способности улавливать временные зависимости и сезонные паттерны

## 2. Среда и воспроизводимость

- Python: 3.8+
- pandas / numpy / scikit-learn: последние стабильные версии
- torch: 2.0+
- Устройство (CPU/GPU): CPU (автоопределение)
- Seed: 42 (фиксирован для random, numpy, torch)
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: часовая
- Горизонт прогноза: 1 час вперед
- Размер датасета: 4322 наблюдения
- Диапазон дат: 2025-01-01 00:00:00 до 2025-06-18 23:00:00
- Есть ли дополнительные внешние признаки: нет, только временная метка
- Комментарий: ряд показывает четкие суточные паттерны с периодом 24 часа, значения варьируются от 80 до 160, отсутствуют явные тренды, заметна сезонность и волатильность в течение дня.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: хронологическое разделение без перемешивания
- Какие доли или интервалы использовались: 70% train, 15% validation, 15% test по времени
- Почему `random split` здесь некорректен: временные ряды имеют автокорреляцию, random split создал бы утечку данных из будущего в прошлое

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: hour, day_of_week, month с циклическим кодированием
- Как обрабатывались пропуски после построения лагов/rolling: строки с NaN удалялись
- Использовалось ли масштабирование и где именно оно обучалось: StandardScaler, обучался только на train данных

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз последнего известного значения
- B2 (`moving-average`): скользящее среднее по окну 24 часа
- B3 (`ridge-lag-features`): Ridge регрессия на лаговых, rolling и календарных признаках
- R1 (`gru-forecast`): GRU сеть с window_size=24, hidden_size=64, 2 слоя

Дополнительно:

- Основная метрика выбора лучшей модели: MAE на validation
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 24 (для захвата суточных паттернов)
- Batch size: 32
- Максимальное число эпох: 50 с early stopping
- Optimizer: Adam
- Learning rate: 0.001 с ReduceLROnPlateau
- Как сохранялась лучшая модель: state_dict() лучшей по validation loss

## 7. Анализ

Наивный baseline B1 показал слабые результаты из-за игнорирования временных паттернов и суточной сезонности. Moving average B2 улучшил результат за счет сглаживания, но все еще уступал моделям с признаками. Ridge на лаговых и rolling признаках B3 значительно превзошел простые baseline, что подтверждает важностьFeature Engineering для временных рядов. GRU модель показала лучшие результаты благодаря способности улавливать сложные временные зависимости и нелинейные паттерны, которые не может линейная модель. Суточная сезонность (24 часа) является ключевым фактором, что подтверждается эффективностью window_size=24. Утечки данных предотвращены за счет корректного temporal split и обучения scaling только на train. Выбор по validation принципиален, так как test должен использоваться только для финальной оценки во избежание overfitting. Основные ошибки моделей приходятся на пиковые значения и периоды высокой волатильности.

## 8. Итоговый вывод

Для такой задачи я бы выбрал GRU модель как базовый подход из-за ее способности улавливать временные зависимости. Простого baseline на признаках достаточно, когда данные имеют относительно простые паттерны и требования к точности не очень высокие. Переход к GRU имеет смысл при сложных нелинейных зависимостях, высокой волатильности и когда требуется максимальная точность прогноза. Главное понимание про корректную валидацию временных рядов — это обязательное использование temporal split вместо random split для предотвращения утечки данных из будущего.
